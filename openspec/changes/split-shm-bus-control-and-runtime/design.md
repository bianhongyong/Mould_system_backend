## Context

当前 `common/comm` 中的 `ShmPubSubBus` 同时承载了两类职责：  

1. 主进程控制面职责（共享内存通道创建、生命周期管理、成员状态治理）；
2. 业务进程数据面职责（通道附着、发布订阅、订阅回调执行）。

该“单类多责任”模型使主进程边界不清晰，且与后续“主进程单线程监督器”方向冲突。当前还存在运行期资源所有权不清的问题：业务进程在对象析构路径里可能触发 `unlink`，与“运行期不释放、仅主进程退出时统一 `unlink`”的策略冲突。

本变更只做总线底座拆分与动态消费者一致性，不实现主进程完整 supervisor（启动编排、重拉起策略、配置全流程接入）。

## Goals / Non-Goals

**Goals:**

- 将共享内存总线拆分为主进程控制面与业务进程数据面两个清晰组件。
- 固化通道命名与生命周期：`launch_plan` 固定后通道名固定为“模块名+通道名”；运行期不执行 `shm_unlink`，仅在主进程最终退出路径由主进程统一 `unlink` 释放。
- 将数据面组件在主进程单例化预初始化，利用 fork 后 COW 机制让子进程复用只读基础状态。
- 建立消费者动态成员协议，覆盖正常退出、异常退出、重建与回收。
- 使用 `(owner_pid, owner_start_epoch)` 标识消费者实例，避免 PID 复用误判。
- 保持 `ModuleBase` 业务接入方式尽量稳定，减少业务模块改造成本。

**Non-Goals:**

- 不实现主进程 supervisor 全流程（优先级启动、指数退避、统一健康检查）。
- 不接入 `launch_plan` 全量启动治理能力。
- 不扩展跨节点通信（如 gRPC 数据面）。
- 不引入 exec 启动路径，仍维持 fork-only 运行模型。

## Decisions

### 决策 1：总线拆分为控制面与数据面双组件

- 方案：新增主进程控制面组件（建议命名 `ShmBusControlPlane`）与业务进程数据面组件（建议命名 `ShmBusRuntime`），原 `ShmPubSubBus` 逐步收敛为兼容层或被替换。
- 理由：主进程与业务进程生命周期、线程模型、责任范围不同，拆分后更易验证和演进。
- 备选：
  - 保留单类并加大量开关：边界继续模糊，难防职责漂移。
  - 在现有类上做“主进程特判”：短期可行，但长期技术债高。

### 决策 2：通道命名固定；运行期不 `unlink`，仅主进程退出时统一释放

- 方案：通道共享内存名由“模块名+通道名”确定；`launch_plan` 一旦确定，通道名集合保持稳定。主进程与子进程在**运行期**均不执行 `shm_unlink`；数据面仅做 attach/use/detach。主进程进入**最终退出**阶段时，在子进程已回收（或已停止附着）之后，对受管通道集合执行一次幂等的 `shm_unlink`。
- 理由：运行期名称与段客体保持稳定，避免中途释放导致子进程重附着失败；退出时由主进程统一释放，满足运维对“干净停机”与重启前清理的期望。
- 备选：
  - 永不 `unlink`：重启或换配置时易遗留同名段，增加一致性负担。
  - 子进程析构时 `unlink`：存在明显误删风险，不可接受。

### 决策 3：消费者实例标识使用 `(owner_pid, owner_start_epoch)`

- 方案：消费者槽位记录 `owner_pid` 与 `owner_start_epoch`，主进程回收/重建时以二元组判定实例身份。
- 理由：仅靠 PID 不能避免系统 PID 复用；二元组可识别“同 PID 不同代实例”。
- 备选：
  - 仅 PID：存在误回收风险。
  - UUID：需要额外跨进程分发与持久化，复杂度更高。

### 决策 4：动态成员变更由控制面驱动，数据面执行本地附着与回调

- 方案：控制面负责在线/离线状态变更与回收推进；数据面在进程内执行订阅 reactor、发布写入和游标推进。
- 理由：成员一致性属于系统治理问题，必须由主进程统一裁决；数据面只处理本地执行效率。

### 决策 5：数据面组件在主进程单例预初始化并通过 COW 复用

- 方案：主进程维护数据面组件单例（`ShmBusRuntimeSingleton`），在 fork 前仅初始化可复用的只读基础状态（配置快照、通道元信息索引）。子进程 fork 后复用该内存页并在本地完成进程特有资源绑定（如 eventfd、reactor 线程、订阅句柄）。
- 理由：通过 COW 复用减少重复初始化和内存占用，同时保留子进程独立运行时状态。
- 约束：
  - fork 前单例不得启动后台线程，不得持有跨 fork 不安全锁状态。
  - fork 后子进程必须执行“后附着初始化”阶段，补齐进程本地资源。
  - 单例中可共享部分必须是只读或按进程惰性复制的数据结构。

### 决策 6：保持 1 进程 = 1 模块 + fork-only

- 方案：每个业务进程只运行一个模块实例，不支持 exec。
- 理由：与现有模块模型一致，简化所有权与生命周期推理，避免 exec 带来的句柄与初始化复杂度。

### 交互时序（异常退出恢复）

```text
Worker(module A) crash
        |
        v
ControlPlane detects child exit
        |
        +--> mark consumer OFFLINE (match by owner_pid + owner_start_epoch)
        |
        +--> trigger reclaim pass for affected channel
        |
        +--> allow next instance to claim new/clean slot
                         |
                         v
               New Worker(module A) attach runtime
                         |
                         v
                  subscribe/publish resume
```

## Risks / Trade-offs

- [风险] 控制面与数据面接口切换期可能出现双路径并存导致行为不一致  
→ 缓解：引入阶段性兼容层，并在测试中固定同一组通道用例同时覆盖新旧路径。
- [风险] 消费者槽位回收逻辑变更可能引入 reclaim 时序回归  
→ 缓解：新增异常退出、PID 复用、高频上下线压力测试，并校验无泄漏与无误回收。
- [风险] 仅退出时 `unlink` 时，若主进程异常崩溃未走到清理路径，可能遗留段客体  
→ 缓解：部署层可提供幂等清理工具；控制面启动时对段头做一致性校验与必要重初始化。
- [风险] 预 fork 单例若错误持有线程/锁状态，会导致子进程继承后行为异常  
→ 缓解：将单例预初始化限制为只读结构，所有线程/FD 在子进程 fork 后再初始化。
- [权衡] 拆分后类数量与接口数量增加，短期理解成本上升  
→ 收益：责任边界清晰，可观测与恢复逻辑更可验证，后续 supervisor 集成成本更低。

## Migration Plan

1. 引入控制面组件与数据面组件骨架，先保持旧接口可编译。
2. 实现主进程数据面单例预初始化路径，限定只读共享状态可进入 fork 前阶段。
3. 将共享内存创建/生命周期职责迁移到控制面，移除数据面 `unlink` 行为；主进程仅在最终退出路径执行统一 `unlink`。
4. 将消费者身份字段扩展为 `(owner_pid, owner_start_epoch)` 并接入成员状态机。
5. 逐步将现有调用点切换到新组件，保留临时适配层。
6. 补齐并通过新增场景测试后，移除旧职责耦合路径。

回滚策略：保留旧总线适配入口，在单次发布窗口内可切回旧路径（仅用于紧急回退，不长期并行）。

## Open Questions

- `owner_start_epoch` 的来源是否由主进程分配单调代数，还是由子进程上报启动时间戳并校验？  
- 槽位重建时是“复用原索引”还是“允许重新分配索引”，对观测指标语义影响如何定义？  
- 控制面是否需要暴露只读通道快照 API 供后续 supervisor 直接复用？
