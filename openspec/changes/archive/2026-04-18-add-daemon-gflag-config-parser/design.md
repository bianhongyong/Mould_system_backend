## Context

程序启动与共享内存分配依赖**全局**信息：所有模块的资源与业务参数、以及各模块的 `input_channel`/`output_channel` 必须在消费方使用前一致可读。策略为**一份总配置** `launch_plan.txt` 描述全部模块的启动参数，**各模块的输入/输出通道单独存放在本模块的 I/O 配置文件**中；总配置内仅保存指向该文件的**路径**。由**解析器**根据**总配置文件路径**与各模块声明的**I/O 配置文件路径**完成加载、校验与聚合。

配置中 `resource`/`module_params` 的键名与工程内 **gflags 已在头文件中声明的名称**一致；**多进程测试程序的主进程**在 `fork` 子进程之前，须根据解析结果将这些键值写入对应的 GFLAG，以验证“配置 → 全局 flag → 多进程继承/可见性”链路（具体继承语义以实现与 gflags 用法为准）。

**本次实现范围不包含守护进程正式程序**：不实现 daemon 主程序及其业务上的子进程拉起与监督；交付解析器及**带 GFLAG 赋值步骤的多进程测试程序**。

## Goals / Non-Goals

**Goals:**

- 定义 `launch_plan.txt` 字典布局（多模块、无数组顶层列表），每模块子字典含 `module_name`、`resource`、`module_params`，以及指向独立 I/O 文件的**路径字段**（实现可选用名如 `io_channels_config_path`，文档与 schema 保持一致即可）。
- 定义**模块 I/O 配置文件**格式：单文件内以字典形式包含该模块的 `input_channel` 与 `output_channel`（仍不使用数组表达通道列表时，沿用既有“命名字典”约定）。
- 定义**路径语义**：
  - **总配置文件路径**：由解析器调用方传入（例如 API 首参 `launch_plan_path`），作为打开 `launch_plan.txt` 的唯一依据。
  - **各模块 I/O 配置文件路径**：写在对应模块子字典的路径字段中；若为相对路径，MUST 相对于**总配置文件所在目录**解析；若为绝对路径，按文件系统绝对路径打开。
- 实现解析器：读取总配置 → 遍历全部模块 → 按路径加载各 I/O 文件 → 全模块校验 → 通道聚合与冲突检测。
- **GFLAG 契约**：`resource`/`module_params` 中每个键名 MUST 对应工程内某翻译单元/头文件中已通过 gflags **注册**的同名 flag；解析器输出键名与之一致且不引入别名。
- **测试程序主进程**：在 `fork` 之前（除非文档约定其它顺序），MUST 根据解析结果对解析到的每个 `resource`/`module_params` 键执行同名 GFLAG 赋值；若某键在 gflags 注册表中不存在，测试 MUST 失败并输出明确错误（防止配置与代码脱节）。
- 提供**多进程测试程序**：主进程完成解析与 GFLAG 赋值后 `fork` 子进程，在父子进程约定的场景下继续验证；并在文档中明确解析器与 GFLAG 在**多进程/fork**下的使用约定。

**Non-Goals:**

- 不实现守护进程正式程序、不接入真实进程启动链路。
- 不实现共享内存实际分配代码，仅输出供分配模块使用的拓扑数据结构。
- **不实现**生产环境守护进程内的 GFLAG 运行期锁定策略（测试程序可仅演示 fork 前赋值；锁定语义若需可后续单独变更）。
- 不设计配置热更新/热重载。

## Decisions

1. **总配置与 I/O 配置拆分**
   - 决策：`launch_plan.txt` 只承载模块级 `module_name`、`resource`、`module_params` 与 I/O 文件路径；`input_channel`/`output_channel` 仅出现在各模块独立 I/O 文件中。
   - 理由：通道与启动参数分文件维护，总表保持精简，路径显式可追溯。
2. **单文件、多模块字典布局（总配置）**
   - 决策：总配置文件内使用嵌套字典表示多模块（例如顶层 `modules` 下以模块标识为键），**不使用数组**。
   - 理由：满足“总配置无数组顶层列表”的约束。
3. **解析器为独立库，守护进程为后续消费者**
   - 决策：本期只交付解析器 API 与数据结构；守护进程将来在适当时机调用解析器并自行完成 GFLAG 与 SHM 流程。
   - 理由：与“不写守护进程正式程序”的范围一致。
4. **键名与 GFLAG 同名，且以代码注册为准**
   - 决策：`resource`/`module_params` 内 key 字符串与头文件/编译单元中 gflags **已注册**的名称一致；多模块若会导致同一 GFLAG 被多模块重复写入且语义冲突，解析或测试阶段须检测并报错（策略在实现中固定并文档化）。
   - 理由：配置即“对已存在 flag 填值”，避免隐式新 flag。
5. **通道聚合在解析器内完成**
   - 决策：解析器在加载全部模块 I/O 文件后，聚合全量 `input_channel`/`output_channel`；跨模块同名通道参数不一致则失败即停。
6. **测试主进程承担 GFLAG 赋值**
   - 决策：测试程序**主进程**在 `fork` 前根据解析结果写入 GFLAG，子进程按测试设计读取 flag 或仅验证解析器其它输出；与后续守护进程“先赋值再 fork”的习惯一致。
   - 理由：满足用户要求的验证路径，且不引入正式 daemon 代码。
7. **失败即停**
   - 决策：任一模块总配置项非法、任一 I/O 文件缺失/不可读/校验失败、通道冲突，或存在无法绑定到已注册 GFLAG 的键，均不视为测试成功。

## Risks / Trade-offs

- **[风险] 相对路径基准目录理解不一致** -> **缓解：** 规范写死为“相对 `launch_plan.txt` 所在目录”；解析器在错误信息中打印解析后的绝对路径。
- **[风险] 配置键遗漏或拼写错误导致 gflags 查无此 flag** -> **缓解：** 测试主进程赋值前校验；失败信息包含模块键与配置键名。
- **[风险] fork 后子进程与父进程共享地址空间拷贝，若误用全局状态可能导致测试与真实集成不一致** -> **缓解：** 在文档中明确约定；测试程序 `waitpid` 回收子进程。
- **[风险] 与后续守护进程集成时出现二次解析** -> **缓解：** 解析器设计为轻量、可重复调用；后续由集成方决定缓存策略。

## Migration Plan

1. 定义 `launch_plan.txt`、模块 I/O 配置文件 schema 与路径字段名；给出相对/绝对路径示例。
2. 实现解析器（总配置 + 按路径加载 I/O）与单元测试。
3. 实现多进程测试程序：主进程解析 → **写 GFLAG** → `fork` → 约定断言；纳入测试套件。
4. 后续由守护进程集成时调用解析器并自行赋值（不在本变更实现）。

回滚策略：停用解析器入口或回退配置文件版本。
