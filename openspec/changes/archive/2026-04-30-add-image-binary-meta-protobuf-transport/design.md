## Context

当前通信链路在底层仅支持共享内存二进制 `payload`，但上层消息类型逐步扩展，模块内存在重复编解码逻辑和通道协议分散定义。现有代码已经具备基础的 payload 编解码接口雏形，但缺少统一的通道类型治理、图片协议约束和 protobuf 演进规范，导致新增类型时容易出现行为漂移与兼容性风险。此次设计的核心约束是保持底层不变，在上层协议与抽象层中建立可扩展、可治理、可验证的统一机制。

## Goals / Non-Goals

**Goals:**
- 建立可插拔的编解码策略层，统一处理 protobuf、图片二进制、原始二进制等上层数据到共享内存二进制 `payload` 的映射。
- 固化默认 protobuf 通道规则，强制通道名与 protobuf 消息名严格一致。
- 定义图片“控制面 protobuf + 数据面编码二进制”的双层协议，并将单消息上限冻结为 3MB。
- 建立总 protobuf 聚合入口和版本演进硬规则，保证跨模块向后兼容。
- 将校验、超限、兼容性等关键失败模式纳入可观测与测试边界。

**Non-Goals:**
- 不改变共享内存 ring/slot 的底层二进制承载方式。
- 不在本次引入 RAW 专用字段（`stride`、`endianness`、`color_space`）强制约束。
- 不在本次支持通道别名映射（alias）或宽松命名匹配。

## Decisions

1. **编解码采用策略模式 + 注册表模式**
   - 决策：为每种 `PayloadType` 提供独立 codec，运行期通过注册表解析和调度。
   - 原因：避免集中式分支判断带来的维护复杂度，允许未来新增类型零侵入扩展。
   - 备选方案：单一大函数 `switch(type)`；被拒绝，原因是扩展成本高、测试隔离差。

2. **默认 protobuf 且通道名与 protobuf 消息名严格一致**
   - 决策：通道未声明时按 protobuf 处理；命名不一致直接视为配置错误。
   - 原因：降低协议治理歧义，避免隐式映射带来的线上排障成本。
   - 备选方案：允许 alias；被拒绝，原因是会引入额外治理表和一致性风险。

3. **图片采用双层传输模型**
   - 决策：`image_bytes` 为 JPEG/PNG/NV12/RAW 等编码二进制；元数据由 `ImageMeta` protobuf 提供。
   - 原因：兼顾性能与可维护性，避免将大图像本体绑定到 protobuf 对象序列化路径。
   - 备选方案：整图放入 protobuf bytes 字段；被拒绝，原因是大图高频场景下解析开销与对象构建成本更高。

4. **图片上限与槽位隔离**
   - 决策：单消息最大 3MB；图片槽位采用独立容量配置。
   - 原因：防止大图占满共享内存资源影响其他通道，保证整体吞吐稳定性。
   - 备选方案：统一槽位配置；被拒绝，原因是小消息场景内存浪费且拥塞风险更高。

5. **protobuf 演进规则硬约束**
   - 决策：字段号只增不改不复用，废弃字段必须 `reserved`，新增字段尽量 `optional`，每次聚合 proto 变更记录 `schema_version`。
   - 原因：保障滚动升级和跨版本模块兼容。
   - 备选方案：仅靠代码评审约束；被拒绝，原因是缺乏制度化检查且长期易失效。

## Risks / Trade-offs

- **[风险] 单图片通道假设与未来多路摄像头需求冲突** → **缓解**：在注册表与规格中预留多通道扩展点，但当前先冻结单通道规则。
- **[风险] 3MB 上限不适配部分高清源** → **缓解**：保留按通道可配置上限能力，同时默认值维持 3MB 并强制告警。
- **[风险] 不引入 RAW 扩展字段可能限制后续格式覆盖** → **缓解**：将 RAW 字段列为后续增量变更，先聚焦当前已知格式。
- **[权衡] 严格同名提升治理清晰度但降低命名灵活性** → **缓解**：通过统一命名规范与自动校验降低迁移成本。
